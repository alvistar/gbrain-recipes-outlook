# Outlook-to-Brain: Email + Directory That Update Your Brain

Emails arrive. Brain pages get smarter. The collector pulls your Outlook
inbox, optionally resolves senders against Azure AD (title, department,
company, manager), and the agent reads the digest to enrich brain pages with
source-attributed context.

## IMPORTANT: Instructions for the Agent

**You are the installer.** Follow these steps precisely.

**The core pattern: code for data, LLMs for judgment.**
Email collection is split into two layers:
1. DETERMINISTIC: the collector script pulls emails, generates Outlook
   links, filters noise, tracks state, detects signatures, checks sent mail,
   and optionally resolves senders against the org directory. Links,
   timestamps, IDs, and classifications are generated by code.
2. LATENT: you (the agent) read the collected digests and make judgment
   calls. Who is important? What entities are mentioned? What action
   items exist? When directory data is available, use it to pre-populate
   people pages with title, department, and company.

**Do not try to pull emails yourself.** Use the collector script. It
handles pagination, deduplication, noise filtering, directory caching,
sent-mail reply detection, and Outlook link generation. If you try to do
this via raw API calls, you WILL miss emails, break pagination, duplicate
messages, or generate incorrect links.

**Default to read-only.** The standard setup uses Microsoft Graph read-only
mail access. Do not request write/send/delete scopes unless the user explicitly
asks for them.

**Collection scope:** normal `collect` is incremental, not a full mailbox
backfill. By default it polls Inbox, Archive, and Sent Items with a safety
lookback window so messages are still captured if the user replies and archives
before the next polling cycle. Subsequent runs process new messages and updated
known messages (for example Inbox -> Archive moves) while state-tracking message
IDs and sent conversation IDs. If the user asks for historical import, implement
an explicit date-range/backfill mode instead of deleting `data/state.json` and
accidentally reprocessing recent mail.

## Architecture

```
Microsoft 365 Account(s)
  |  (read-only OAuth via device code flow)
  v
@softeria/ms-365-mcp-server (handles auth, token refresh, Graph API)
  |  (MCP protocol via @modelcontextprotocol/sdk)
  v
Outlook Collector Script (deterministic Node.js)
  |  Calls: list-mail-folder-messages, list-mail-folders, list-users
  |  Default folders: Inbox + Archive + Sent Items, with lookback overlap
  |  Handles: pagination, dedup, sent-mail checks, folder moves, noise filtering,
  |           signature detection, directory cache
  v  Outputs:
  +-- data/runs/{YYYY-MM-DD}/{RUN_ID}.json (per-collection delta)
  +-- data/messages/{YYYY-MM-DD}.json      (cumulative daily aggregate)
  +-- data/directory/{hash}.json           (cached directory lookups, 7d TTL)
  +-- data/digests/{YYYY-MM-DD}.md         (markdown digest for agent)
  +-- data/state.json                      (pagination state, known IDs/folders)
  |
  v
Agent reads digest
  |  Judgment calls:
  +-- Entity detection (people, companies, projects mentioned)
  +-- Brain page updates (timeline section, compiled truth)
  +-- People page creation with directory metadata when available
  +-- Action item extraction
  +-- Priority classification (urgent / normal / noise)
```

## Opinionated Defaults

**Mail-only mode (default):** authenticate with `--read-only`, collect mail,
filter noise, generate Outlook links, and create/update Brain pages. This works
for work/school and personal Outlook accounts and does not require Azure admin
consent.

**Directory-enriched mode:** add `--org-mode`, `MS365_MCP_CLIENT_ID`,
`MS365_MCP_TENANT_ID`, and delegated `User.Read.All` admin consent. The digest
then includes title, department, company, and manager where Microsoft Graph
returns them.

**Digest format:** Daily markdown with sections:
- Signatures pending (DocuSign etc. needing action)
- Messages to triage (real emails from real people)
- Noise (filtered, available if needed)

Every email gets a baked-in Outlook link generated by code. Prefer Graph
`webLink`; otherwise construct the correct OWA/deep link from the message ID and
account type. Do not ask the LLM to invent Outlook URLs.

1. **GBrain installed and configured** (`gbrain doctor` passes)

## Setup Flow

### Step 1: Authenticate with Microsoft 365

Ask the user: "Do you want mail-only read-only mode, or org-mode directory
enrichment? Mail-only needs only device-code login. Org-mode needs Azure AD app
registration and admin consent for directory lookup."

#### Option A: Mail-only read-only mode (recommended default)

Use this for most installs, and especially when the user wants least privilege.

```bash
# Optional but recommended in persistent environments:
mkdir -p /opt/data/ms365-mcp
chmod 700 /opt/data/ms365-mcp
export MS365_MCP_TOKEN_CACHE_PATH=/opt/data/ms365-mcp/token-cache.json
export MS365_MCP_SELECTED_ACCOUNT_PATH=/opt/data/ms365-mcp/selected-account.json

npx @softeria/ms-365-mcp-server --read-only --login
```

Validate by parsing JSON, not just exit code:

```bash
npx @softeria/ms-365-mcp-server --read-only --verify-login
```

The command may exit zero even when the JSON says `{"success": false}`. Treat
`success: true` as the source of truth.

**STOP until read-only authentication validates.**

#### Option B: Work/School org-mode directory enrichment

Tell the user:
"I need Azure AD app registration credentials for Microsoft Graph directory
access.

1. Go to https://portal.azure.com
2. Navigate to **Azure Active Directory** > **App registrations** > **New registration**
3. Name: 'GBrain Outlook Collector' (anything works)
4. Supported account types: **Accounts in this organizational directory only**
5. Redirect URI: leave blank (we use device code flow)
6. Click **Register**
7. Copy the **Application (client) ID** and **Directory (tenant) ID** from the Overview page
8. Go to **API permissions** > **Add a permission** > **Microsoft Graph** > **Delegated permissions**
9. Add: `Mail.Read`, `User.Read.All`
10. Click **Grant admin consent** (requires admin privileges)"

Then authenticate the MCP server:

```bash
export MS365_MCP_CLIENT_ID=<client-id>
export MS365_MCP_TENANT_ID=<tenant-id>
export MS365_MCP_TOKEN_CACHE_PATH=/opt/data/ms365-mcp/token-cache.json
export MS365_MCP_SELECTED_ACCOUNT_PATH=/opt/data/ms365-mcp/selected-account.json
npx @softeria/ms-365-mcp-server --read-only --login --org-mode
```

**STOP until authentication validates.** Directory lookup can still fail if
`User.Read.All` admin consent is missing; mail collection should continue.

### Step 2: Install the Collector

```bash
cd /path/to/gbrain-recipes-outlook
npm install
```

If `npm install` fails on the Microsoft 365 MCP peer dependency, check the
published package version and use the repository's compatible 0.x range. The
collector has been validated with `@softeria/ms-365-mcp-server` 0.107.x.

### Step 3: Run First Collection

```bash
export MS365_MCP_TOKEN_CACHE_PATH=/opt/data/ms365-mcp/token-cache.json
export MS365_MCP_SELECTED_ACCOUNT_PATH=/opt/data/ms365-mcp/selected-account.json
# Default: Inbox + Archive + Sent Items, 2h overlap.
node outlook-collector/collector.mjs collect --folders inbox,archive,sentitems --lookback 2h
node outlook-collector/collector.mjs digest
```

Verify: `ls data/digests/` should show today's digest file. Read the digest.
Confirm it contains real emails with working Outlook links. Run collect twice in
one minute and verify the second run does not duplicate messages. Also verify
`data/runs/YYYY-MM-DD/` contains one file per run and
`data/messages/YYYY-MM-DD.json` remains cumulative rather than being replaced by
an empty later run.

### Step 4: Enrich Brain Pages

This is YOUR job (the agent). Read the digest and the structured JSON. For each
email:

1. **Detect entities**: who sent it? Who is mentioned? What companies or projects?
2. **Use directory data**: if the digest shows `(VP Engineering @ Acme Corp, Engineering)`
   next to a sender, use that to pre-populate the people page:
   ```markdown
   ---
   type: person
   name: Alice Smith
   email: alice@acme.com
   title: VP of Engineering
   department: Engineering
   company: Acme Corp
   source: outlook
   tags: [outlook, person]
   ---
   ```
3. **Check the brain**: `gbrain search "sender name"` to check for existing pages.
4. **Update brain pages**: append an `## Outlook timeline` section entry:
   `- YYYY-MM-DD | Email from {sender}: {subject} [Source: Outlook, {date}]({link})`
5. **Create new pages**: if sender/entity/project is notable and has no page,
   create one using `gbrain put` (use directory data when available).
6. **Extract action items**: if the email requires a response or action, log it
   on an appropriate project/person/action page with a source link.
7. **Optional first-class timeline**: if the brain uses timeline rows, also run
   `gbrain timeline-add` for important dated events. Otherwise the timeline
   entry in compiled truth is acceptable.
8. **Sync/index**: if writing markdown files directly, run
   `gbrain sync --no-pull` to index and embed them. If using `gbrain put`,
   embeddings happen during the write. Either way, verify with
   `gbrain search "sender name"` to confirm pages are findable.

### Step 5: Set Up Cron

The collector should run every 30 minutes:

```bash
*/30 * * * * cd /path/to/gbrain-recipes-outlook && MS365_MCP_TOKEN_CACHE_PATH=/opt/data/ms365-mcp/token-cache.json MS365_MCP_SELECTED_ACCOUNT_PATH=/opt/data/ms365-mcp/selected-account.json node outlook-collector/collector.mjs collect && node outlook-collector/collector.mjs digest
```

The agent should read the digest on a schedule (e.g., 3x/day) and run the
enrichment flow from Step 4.

## Implementation Guide

These mirror the production-tested Gmail-to-Brain collector rules, adapted for
Microsoft 365.

### Collector Capabilities

The collector script must provide:
1. **collect** — pull Outlook messages via MCP/Graph, deduplicate by immutable
   message ID, store structured JSON, and bake in Outlook links.
2. **digest** — generate a markdown digest from collected emails grouped by
   signatures pending, messages to triage, and noise.
3. **state tracking** — remember last collection timestamp and known message IDs
   to avoid re-processing.
4. **sent-mail detection** — pull/inspect sent messages to determine which
   inbound threads already have replies.
5. **directory cache** — for org-mode only, cache sender lookups for 7 days.

### Noise Filtering (Deterministic)

The collector filters these sender patterns as noise:
- `noreply`, `no-reply`, `notifications@`, `calendar-notification`
- `mailer-daemon`, `postmaster`, `donotreply`

Use simple substring matching on normalized sender addresses. Do not rely on the
LLM to decide whether obvious notification mail is noise.

### Signature Detection

Signature-related emails are flagged separately:
- DocuSign, Dropbox Sign, HelloSign, PandaDoc
- "please sign", "signature needed", "ready for your signature"

Test both sender and subject. Signature requests often come from service sender
addresses, not only from subject text.

### Outlook Link Generation (CRITICAL)

Prefer Microsoft Graph `webLink` when present. If absent, construct an Outlook
web link from the immutable message ID and account type. Links are generated in
CODE, never by the LLM. Every digest entry must include an `[Open in Outlook]`
link that opens the correct message for the authenticated account.

### Deduplication and Lookback Collection

```pseudo
collect(folders = ['inbox', 'archive', 'sentitems'], lookback = '2h'):
  state = load_state()
  since = state.lastCollect - lookback

  for folder in folders:
    date_field = folder == 'sentitems' ? 'sentDateTime' : 'receivedDateTime'
    messages = graph.list_mail_folder_messages(folder_id(folder), date_field >= since)
    for msg in messages:
      existing = state.knownIds[msg.id]
      update_state(msg.id, folder, msg.changeKey, msg.lastModifiedDateTime)
      if folder == 'sentitems':
        sentThreadIds.add(msg.conversationId)
      else if !existing or folder/changeKey/lastModified changed:
        emit msg for digest/enrichment
```

Use `mailFolderId` (well-known folder IDs or IDs resolved from `list-mail-folders`),
not display names alone. Folder display names are localized (`Posta in arrivo`,
`Posta inviata`, etc.).

**Why Inbox + Archive matters:** If the user replies and archives an email before
the next 10-minute polling cycle, Inbox-only polling can miss it. The Archive
lookback catches the original message after it moves.

**Why Sent Items matters:** Without it, the digest can show "awaiting response"
on threads you already replied to. Sent mail acts as a negative filter.

### Directory Lookup

For work/school accounts with `--org-mode`, the collector resolves each unique
sender against Azure AD using an OData filter (`$filter=mail eq '...'`). Email
addresses must be escaped for OData (single quotes doubled) to prevent filter
injection via crafted sender addresses. Results are cached for 7 days. The digest includes the sender's title, department, and company when
available.

Use `--refresh-directory` to force re-fetch all cached directory entries.
Personal accounts skip directory lookup entirely (no Azure AD available).

Directory lookup is best-effort. If `list-users` fails with insufficient
privileges, log a warning and continue mail collection.

## What the Agent Should Test After Setup

1. **Read-only auth:** `--read-only --verify-login` returns JSON with
   `success: true`. Do not trust shell exit status alone.
2. **Outlook links:** Open at least one digest link and verify it opens the
   correct message/account.
3. **Deduplication:** Run collect twice in one minute. Verify no duplicate
   messages in today's JSON/digest.
4. **Noise filtering:** Verify obvious `noreply`/`notifications@` messages land
   in noise, not triage.
5. **Signature detection:** Verify DocuSign/Dropbox Sign/etc. land in the
   signatures section.
6. **Sent mail:** Reply to a test email manually, run collect, and verify the
   thread is marked as replied/does not remain awaiting response.
7. **Directory degradation:** In mail-only mode or without `User.Read.All`,
   directory lookup should warn and continue; mail collection must still work.
8. **GBrain enrichment:** Create/update at least one person page and verify
   `gbrain search "sender name"` finds it with embeddings.

## Troubleshooting

**No emails collected:**
- Run `npx @softeria/ms-365-mcp-server --read-only --login` to re-authenticate.
- Validate with `--read-only --verify-login` and parse JSON `success`.
- Ensure persistent token env vars point at existing files:
  - `MS365_MCP_TOKEN_CACHE_PATH`
  - `MS365_MCP_SELECTED_ACCOUNT_PATH`
- Check the collector's stderr output for error messages.

**Login works manually but collector says "No accounts found":**
- The MCP stdio transport may be dropping custom env vars. Ensure the collector
  passes the parent env to `StdioClientTransport` and that token cache paths are
  exported before running `collect`.

**Directory lookup not working:**
- Requires `--org-mode` (work/school accounts only).
- Requires `User.Read.All` delegated permission and admin consent.
- If unavailable, continue in mail-only mode; people pages simply won't get
  automatic title/department/company metadata.

**Outlook links don't work:**
- Links open in the default browser. The user must be logged into the correct
  Microsoft account.
- Prefer Graph `webLink`; if constructing links, include the encoded message ID
  exactly as returned by Graph.

**Digest is empty but collection ran:**
- Check `data/messages/` for JSON files.
- All emails might be filtered as noise; check the noise section.
- The state file may already know all recent message IDs.

**Corrupted state (collector keeps re-collecting old emails):**
- Back up and delete `data/state.json` to start fresh. The collector will
  re-collect recent emails, but brain-side deduplication should catch duplicates.
- Do not use state deletion as a substitute for a real historical backfill.
